## 1. 项目是什么？—— 一句话本质

**Understand Anything** 是一个 **AI 驱动的代码库理解工具**，它的核心工作流程是：

```
你的代码库 → Tree-sitter 静态分析 + LLM 语义理解 → 知识图谱 (JSON) → 交互式 Dashboard 可视化
```

简单来说，它把一个项目的所有文件、函数、类、依赖关系"翻译"成一张**可搜索、可点击、可探索的图**，让你不再盲目地一行行读代码，而是先看到全局结构再深入细节。

### 知识点：为什么是"知识图谱"而不是"文档"？

**是什么？**
知识图谱用**节点（Node）**和**边（Edge）**来表示实体及其关系。在本项目中：
- **节点** = 文件、函数、类、配置、服务、数据库表等
- **边** = import 调用、继承、包含、依赖等关系

**为什么？**
传统文档是线性的（README → 按文件夹浏览），而代码库的关系是**网状的**（A 调用 B，B 继承 C，C 配置了 D）。线性文档无法表达这种网络拓扑。

**追问：为什么不用 UML 图或调用图工具？**
- UML 工具需要手动维护，过时快
- 传统调用图工具只展示结构关系（谁调用了谁），缺少**语义信息**（这个类是干什么的？）
- Understand Anything 的创新在于：**Tree-sitter 提供确定性结构 + LLM 补充语义摘要**，两者结合

## 2. 为什么需要它？—— 解决的真实问题

想象你刚加入一个团队，面对一个 **20 万行代码**的项目：

| 场景 | 传统方式 | Understand Anything |
|------|---------|---------------------|
| 了解项目结构 | 逐个打开文件夹 | 一张图看到全部文件和关系 |
| 找某个功能在哪 | `grep` 搜索 + 猜测 | 语义搜索 "哪些部分处理认证？" |
| 理解业务逻辑 | 问同事 + 读 README | Domain View 展示业务域和流程 |
| Code Review | 只看 diff 的变更文件 | Diff Impact 分析受影响的完整链路 |
| 新人 Onboarding | 口头传授 + 散落文档 | Guided Tour 按依赖顺序引导学习 |

## 5. 核心技术栈详解

### 5.1 后端/引擎层

| 技术 | 版本 | 用途 | 为什么选它？ |
|------|------|------|-------------|
| **TypeScript** | 5.7+ | 全项目统一语言 | 类型安全 + 严格模式，捕获运行时错误于编译期 |
| **Node.js** | >= 22 (v24 开发) | 运行时环境 | 最新 API + 原生 ESM 支持 |
| **pnpm** | 10.6+ | 包管理器 | 高效磁盘利用 + monorepo workspace 原生支持 |
| **web-tree-sitter** | 0.26+ | 语法解析（WASM） | 为什么不原生 tree-sitter？→ 原生绑定在 darwin/arm64 + Node 24 上失败；WASM 跨平台一致 |
| **Zod** | 4.3+ | Schema 验证 | 运行时类型校验 + 自动修复脏数据（LLM 输出不总是完美的） |
| **Fuse.js** | 7.1+ | 模糊搜索 | 轻量、零依赖、容错匹配（用户拼错也能搜到） |
| **Vitest** | 3.1+ | 测试框架 | 与 Vite 生态一致 + 极快的 HMR |

### 5.2 前端/Dashboard 层

| 技术 | 版本 | 用途 | 为什么选它？ |
|------|------|------|-------------|
| **React** | 19 | UI 框架 | 最新版本 + 并发特性 + Server Components 准备 |
| **@xyflow/react** | 12 | 图可视化 | React 生态最成熟的图库，支持自定义节点/边/布局 |
| **Zustand** | 5 | 状态管理 | 为什么不用 Redux？→ 更轻量、更少的样板代码、内置 DevTools |
| **TailwindCSS v4** | 4 | CSS 框架 | 为什么 v4？→ CSS-first 配置（无需 tailwind.config.js）、OXIDE 引擎更快 |
| **Vite** | 6.4+ | 构建工具 | 极速 HMR + 原生 ESM + 内置中间件（dev server 直接 serve JSON） |
| **prism-react-renderer** | 2.4+ | 代码高亮 | 轻量、React 原生、支持 40+ 语言 |
| **ELK.js** | 0.9+ | 层次布局算法 | 处理有向图的层次化排列（适合展示架构层级） |
| **d3-force** | 3.0+ | 力导向布局 | 处理无向/混合图的物理模拟布局 |
| **graphology** | 0.25+ | 图数据结构 | 纯 JS 图库 + Louvain 社区发现算法 |
| **react-markdown** | 10.1+ | Markdown 渲染 | 渲染 LLM 生成的摘要（支持 GFM） |

### 5.3 关键设计约束

| 约束 | 原因 |
|------|------|
| **ESM only** (`"type": "module"`) | 现代 JavaScript 标准，避免 CJS/ESM 互操作地狱 |
| **Dashboard 只导入 core 的子路径导出** (`./search`, `./types`, `./schema`) | 避免将 Node.js 模块（fs, path 等）打包进浏览器 |
| **tree-sitter 用 WASM 而非 native binding** | 兼容性：native binding 在 Apple Silicon + Node 24 上编译失败 |
| **Monorepo (pnpm workspaces)** | core 和 dashboard 共享类型，但可以独立发布 |

## 6. 核心模块一：`packages/core` —— 分析引擎

这是整个项目的**心脏**，是一个纯 TypeScript 库，没有任何 UI 依赖，可以在 Node.js 和浏览器环境中运行（通过子路径导出）。

### 6.1 数据模型（知识图谱的"词汇表"）

**文件**: [types.ts](understand-anything-plugin/packages/core/src/types.ts)

#### 节点类型（21 种，分 4 大类）

```typescript
// 代码类（5 种）—— 来自源代码的直接产物
type CodeNode = "file" | "function" | "class" | "module" | "concept"

// 非代码基础设施（8 种）—— 来自配置/部署/数据文件
type InfraNode = "config" | "document" | "service" | "table" | "endpoint"
              | "pipeline" | "schema" | "resource"

// 业务域（3 种）—— 来自 /understand-domain 命令
type DomainNode = "domain" | "flow" | "step"

// 知识库（5 种）—— 来自 /understand-knowledge 命令
type KnowledgeNode = "article" | "entity" | "topic" | "claim" | "source"
```

#### 边类型（35 种，分 8 大类）

| 类别 | 边类型 | 含义 |
|------|--------|------|
| **Structural** | `imports`, `exports`, `contains`, `inherits`, `implements` | 代码结构关系 |
| **Behavioral** | `calls`, `subscribes`, `publishes`, `middleware` | 运行时行为 |
| **Data flow** | `reads_from`, `writes_to`, `transforms`, `validates` | 数据流向 |
| **Dependencies** | `depends_on`, `tested_by`, `configures` | 依赖关系 |
| **Semantic** | `related`, `similar_to` | 语义关联 |
| **Infrastructure** | `deploys`, `serves`, `provisions`, `triggers` | 部署/运维 |
| **Schema/Data** | `migrates`, `documents`, `routes`, `defines_schema` | 模式定义 |
| **Domain** | `contains_flow`, `flow_step`, `cross_domain` | 业务流程 |
| **Knowledge** | `cites`, `contradicts`, `builds_on`, ... | 知识推理 |

#### 核心数据接口

```typescript
interface GraphNode {
  id: string;                // 唯一标识，如 "file:src/App.tsx"
  type: NodeType;            // 21 种之一
  name: string;              // 显示名称
  filePath?: string;         // 源文件路径
  lineRange?: [number, number]; // 代码位置 [起始行, 结束行]
  summary: string;           // LLM 生成的自然语言摘要
  tags: string[];            // 标签（如 ["auth", "api", "rest"]）
  complexity: "simple" | "moderate" | "complex"; // 复杂度
}

interface GraphEdge {
  source: string;            // 起点 node ID
  target: string;            // 终点 node ID
  type: EdgeType;            // 35 种边类型之一
  direction: "forward" | "backward" | "bidirectional";
  weight: number;            // 0-1 权重（边的"强度"）
}
```

### 6.2 Schema 验证与自动修复

**文件**: [schema.ts](understand-anything-plugin/packages/core/src/schema.ts)

这是一个**非常关键的模块**。因为知识图谱的数据来自 LLM 输出，而 LLM 不总是严格遵守格式要求，所以需要一个**宽容但严格的验证管道**：

```
原始 LLM 输出 → sanitizeGraph() → normalizeGraph() → autoFixGraph() → validateGraph()
     清洗空值       映射别名          填充默认值          Zod 严格校验
```

**四层验证策略**：

| 层级 | 操作 | 示例 |
|------|------|------|
| Tier 1 - Sanitize | 清洗空值、小写枚举 | `null` tags → `[]`，`"FILE"` → `"file"` |
| Tier 2 - Auto-fix | 填充缺失字段、类型强制转换 | 缺失 complexity → `"moderate"`，字符串 weight → 数字 |
| Tier 3 - Validate & Drop | Zod 校验，丢弃无法修复的数据 | 无效节点/边被移除并记录 issue |
| Tier 4 - Fatal | 致命错误阻断 | 缺少 project 元数据 → 整体验证失败 |

**别名映射系统**：LLM 经常用不同词汇表达同一概念（如 `func` vs `function`，`invoke` vs `calls`），schema 维护了一个完整的别名映射表（`NODE_TYPE_ALIASES`, `EDGE_TYPE_ALIASES`），在验证前自动归一化。

### 6.3 Tree-sitter 插件体系

**文件**: [plugins/tree-sitter-plugin.ts](understand-anything-plugin/packages/core/src/plugins/tree-sitter-plugin.ts)

Tree-sitter 是一个**增量解析器**，能将源代码解析为具体的语法树（Concrete Syntax Tree, CST）。本项目使用 `web-tree-sitter`（WASM 版本）而非原生绑定。

**支持的编程语言**（通过 tree-sitter grammars）：
- TypeScript/JavaScript, Python, Go, Rust, Java, C/C++, C#, Ruby, PHP

**AST 提取器**（[plugins/extractors/](understand-anything-plugin/packages/core/src/plugins/extractors/)）：
每个提取器负责从特定语言的 CST 中提取结构化信息：

```typescript
interface StructuralAnalysis {
  functions: Array<{ name, lineRange, params, returnType }>;
  classes: Array<{ name, lineRange, methods, properties }>;
  imports: Array<{ source, specifiers, lineNumber }>;
  exports: Array<{ name, lineNumber, isDefault }>;
  // 非代码文件的扩展信息
  sections?: SectionInfo[];      // Markdown 标题层级
  definitions?: DefinitionInfo[]; // SQL 表/Proto 消息/YAML 键
  services?: ServiceInfo[];       // Docker 服务/K8s 部署
  endpoints?: EndpointInfo[];     // API 路由定义
}
```

**非代码解析器**（[plugins/parsers/](understand-anything-plugin/packages/core/src/plugins/parsers/)）：
针对非源代码文件的专用解析器：YAML, JSON, TOML, Env, Dockerfile, SQL, GraphQL, Protobuf, Terraform, Makefile, Shell, Markdown

### 6.4 搜索引擎

**文件**: [search.ts](understand-anything-plugin/packages/core/src/search.ts)

两层搜索策略：

| 模式 | 实现 | 适用场景 |
|------|------|---------|
| **模糊搜索** | Fuse.js | 按名称搜索（"找 AuthController"） |
| **语义搜索** | embedding-search.ts (余弦相似度) | 按含义搜索（"哪些部分处理认证？"） |

### 6.5 增量更新机制

**文件**: [fingerprint.ts](understand-anything-plugin/packages/core/src/fingerprint.ts), [staleness.ts](understand-anything-plugin/packages/core/src/staleness.ts), [change-classifier.ts](understand-anything-plugin/packages/core/src/change-classifier.ts)

这是性能优化的核心。对于大型代码库，每次都重新分析所有文件是不现实的。

**工作原理**：
1. **基线建立**：首次分析后，为每个文件生成**结构指纹**（函数签名列表、类定义列表、import 列表的 hash）
2. **变更检测**：下次运行时，对比当前文件指纹与基线
3. **变更分类**：
   - `UNCHANGED` → 跳过
   - `STRUCTURAL` → 函数/类/import 发生变化 → 需要 LLM 重新分析
   - `FULL_UPDATE` → 文件新增或完全重写 → 完整重新分析
4. **图合并**：删除旧节点 → 插入新节点 → 重算受影响的边

## 8. 核心 Skill/Agent 系统 —— 多智能体流水线

这是整个项目最独特的部分：**用 LLM Agent 协作完成代码分析**。

### 8.1 Skill 是什么？

Skill 是 AI 编程助手（Claude Code, Cursor 等）中的**可复用命令模板**。类似于 Unix 的 shell script，但执行者是 LLM 而不是 shell。

本项目定义了 8 个 Skill：

| Skill | 命令 | 功能 |
|-------|------|------|
| `understand` | `/understand` | **核心**：完整分析流水线，生成 knowledge-graph.json |
| `understand-dashboard` | `/understand-dashboard` | 启动本地 Dashboard 开发服务器 |
| `understand-chat` | `/understand-chat <question>` | 对话式问答（基于已生成的图谱） |
| `understand-diff` | `/understand-diff` | 分析当前 git 变更的影响范围 |
| `understand-explain` | `/understand-explain <file>` | 深度解释单个文件或函数 |
| `understand-onboard` | `/understand-onboard` | 为新成员生成入门指南 |
| `understand-domain` | `/understand-domain` | 提取业务领域知识（域/流程/步骤） |
| `understand-knowledge` | `/understand-knowledge <path>` | 分析 Karpathy-style wiki 知识库 |

### 8.2 Agent 是什么？

Agent 是 **LLM 子代理的提示词模板**。主 Skill 将任务分解后，dispatch 给专门的 Agent 并行处理。

| Agent | 角色 | 输入 | 输出 |
|-------|------|------|------|
| **project-scanner** | 项目清点员 | 项目目录 | `scan-result.json`（文件列表 + 语言 + 框架 + import map） |
| **file-analyzer** | 文件分析师 |一批文件（20-30 个） | `batch-N.json`（节点 + 边），**最多 5 个并行** |
| **architecture-analyzer** | 架构师 | 全部文件节点 + import 边 | `layers.json`（架构层级分配） |
| **tour-builder** | 导游 | 全部节点 + 边 + 层级 | `tour.json`（引导式学习路径） |
| **assemble-reviewer** | 组装审查员 | 合并后的图 + batch 文件 | `assemble-review.json`（审查意见） |
| **graph-reviewer** | 图审计员（可选） | 最终图 | `review.json`（完整性检查） |
| **domain-analyzer** | 业务分析师 | 代码结构 | domain-graph.json（业务域/流程/步骤） |
| **article-analyzer** | 文章分析师 | Wiki 文章 | 实体/声明/关系抽取 |

### 8.3 `/understand` 的 7 阶段流水线

**文件**: [skills/understand/SKILL.md](understand-anything-plugin/skills/understand/SKILL.md)（约 850 行，**最重要的文件**）

```
Phase 0: Pre-flight（预检）
  ├── 解析参数（路径、--full、--language 等）
  ├── Git worktree 检测与重定向
  ├── 确保 plugin 已构建
  ├── 收集项目上下文（README, manifest, 目录树, 入口文件）
  ├── 语言配置（自动检测 or 用户指定）
  └── 决定：全量分析 or 增量更新 or 跳过

Phase 0.5: Ignore Configuration（忽略配置）
  └── 生成/确认 .understandignore 文件

Phase 1: SCAN（扫描）[仅全量]
  └── dispatch project-scanner Agent → scan-result.json

Phase 1.5: BATCH（分批）
  └── compute-batches.mjs → batches.json（语义分组 + import 邻居映射）

Phase 2: ANALYZE（分析）
  ├── [全量] dispatch file-analyzer Agent × N（最多 5 并发）
  ├── [增量] 只分析变更文件
  └── merge-batch-graphs.py → assembled-graph.json（合并 + 去重 + 标准化）

Phase 3: ASSEMBLE REVIEW（组装审查）
  └── dispatch assemble-reviewer Agent → assemble-review.json

Phase 4: ARCHITECTURE（架构分层）
  ├── 注入语言上下文（languages/*.md）
  ├── 注入框架上下文（frameworks/*.md）
  ├── 注入 locale 指南（locales/*.md）
  └── dispatch architecture-analyzer Agent → layers.json

Phase 5: TOUR（导览构建）
  └── dispatch tour-builder Agent → tour.json

Phase 6: REVIEW（审查）
  ├── [默认] inline 确定性验证（Node.js 脚本）
  └── [--review] dispatch graph-reviewer Agent（完整 LLM 审查）

Phase 7: SAVE（保存）
  ├── 写入 knowledge-graph.json
  ├── 生成指纹基线（build-fingerprints.mjs）
  ├── 写入 meta.json
  ├── 清理中间文件
  └── 输出最终报告
```

## 9. 数据流全链路追踪 —— 从 `/understand` 命令到知识图谱

让我们用一个具体例子追踪数据如何流动：

假设用户在 `my-project/` 目录下运行 `/understand`：

```
1️⃣ 用户输入: /understand
      ↓
2️⃣ Phase 0: 解析参数
   PROJECT_ROOT = "/path/to/my-project"
   读取 README.md, package.json, 目录树
      ↓
3️⃣ Phase 1: project-scanner Agent
   运行 scan-project.mjs → 发现 247 个文件
   运行 extract-import-map.mjs → 解析出 487 条 import 关系
   输出: scan-result.json { name, files[], languages[], frameworks[], importMap{} }
      ↓
4️⃣ Phase 1.5: compute-batches.mjs
   输入: 247 个文件 + import map
   输出: batches.json [
     { batchIndex: 0, files: [...20 files], batchImportData: {...}, neighborMap: {...} },
     { batchIndex: 1, files: [...20 files], ... },
     ... 共 13 个 batch
   ]
   ★ 关键优化：语义分组确保相关文件在同一 batch，邻居映射帮助跨 batch 边的发现
      ↓
5️⃣ Phase 2: file-analyzer Agent × 5 并发
   Batch 0 Agent 收到 20 个文件 + pre-resolved imports + neighbor map
   对每个文件:
     a. 运行 extract-structure.mjs (Tree-sitter 解析 → AST → functions/classes/imports)
     b. LLM 阅读 AST + 源代码 → 生成 summary/tags/complexity + edges
   输出: batch-0.json { nodes: [...], edges: [...] }
   ... (重复 13 次，每次 5 个并发)
      ↓
6️⃣ merge-batch-graphs.py
   读取 batch-0.json ~ batch-12.json
   合并: 289 nodes + 1,847 edges
   标准化: ID 前缀修正、复杂度别名映射、去重
   tested_by linker: 修复反转/无效的测试覆盖边
   输出: assembled-graph.json
      ↓
7️⃣ Phase 4: architecture-analyzer Agent
   输入: 289 个文件节点 + 全部 imports 边 + 目录树 + 语言/框架上下文
   LLM 分析 → 分配 5 个层级:
     - layer:api ("API Layer") → [file:src/routes/*, endpoint:*]
     - layer:service ("Business Logic") → [file:src/services/*]
     - layer:data ("Data Access") → [file:src/models/*, table:*]
     - layer:config ("Configuration") → [config:*, document:README.md]
     - layer:infra ("Infrastructure") → [service:Dockerfile, pipeline:.github/*]
   输出: layers.json
      ↓
8️⃣ Phase 5: tour-builder Agent
   输入: 全部节点 + 边 + 层级 + README 内容
   LLM 生成 8 步导览:
     Step 1: "Project Overview" → [document:README.md]
     Step 2: "Entry Point" → [file:src/index.ts]
     Step 3: "API Routes" → [file:src/routes/index.ts, endpoint:*]
     ...
   输出: tour.json
      ↓
9️⃣ Phase 6: 验证
   inline validate.cjs 脚本:
     - 检查必填字段 ✓
     - 检查引用完整性（每条边的 source/target 都存在）✓
     - 检查孤儿节点 ✗ 3 个（记录为 warnings）
     - 检查层级覆盖率 ✓
   输出: review.json { issues: [], warnings: [...], stats: {...} }
      ↓
🔟 Phase 7: 保存
   写入: .understand-anything/knowledge-graph.json (version: "1.0.0", 289 nodes, 1847 edges, 5 layers, 8 tour steps)
   写入: .understand-anything/meta.json (commitHash, analyzedAt, version)
   生成: fingerprints baseline (用于下次增量更新)
   清理: intermediate/ 目录（保留 scan-result.json）

🪙 最终产出:
   my-project/.understand-anything/
   ├── knowledge-graph.json   ← 🎯 核心产物
   ├── meta.json
   └── config.json           ← autoUpdate, outputLanguage
```

## 10. 关键设计模式与架构决策

### 10.1 确定性 + 语义性的混合架构

**这是什么？**
- **确定性部分**（Tree-sitter）：相同的输入永远产生相同的输出 → 可复现、可缓存
- **语义性部分**（LLM）：理解代码意图 → 人类可读的摘要、标签、层级

**为什么这样设计？**
如果全用 LLM → 慢、贵、不确定（同一个文件两次分析可能产生不同的结构）
如果全用静态分析 → 无法理解"这个类是干什么的"，只能知道"它有 3 个方法"
两者互补：静态分析提供**准确的结构骨架**，LLM 提供**有意义的血肉**

### 10.2 Agent 并行化 + 批处理

**这是什么？**
- 文件分析不是逐个进行，而是按 **20-30 个文件一批**
- 最多 **5 个 Agent 同时运行**

**为什么？**
- 减少 LLM 调用次数（一次 prompt 分析 20 个文件 vs 20 次 prompt 各分析 1 个）
- 通过批内的 import 预解析（`batchImportData`），Agent 不需要自己重新解析 import
- 通过 `neighborMap`（跨批次邻居），即使文件在不同 batch 也能发现跨 batch 的边

### 10.3 子路径导出（Browser-safe Exports）

**这是什么？**
core 包的 `package.json` 定义了子路径导出：

```json
"exports": {
  ".": "./dist/index.js",        // 完整导出（含 Node.js 模块）
  "./search": "./dist/search.js",  // 浏览器安全（只有搜索功能）
  "./types": "./dist/types.js",   // 浏览器安全（只有类型）
  "./schema": "./dist/schema.js"  // 浏览器安全（只有验证）
}
```

**为什么？**
Dashboard 运行在浏览器中，不能引入 `fs`, `path`, `child_process` 等 Node.js 模块。通过子路径导出，Dashboard 只导入需要的部分。

### 10.4 LLM 输出的防御性处理

**这是什么？**
schema.ts 实现了完整的 **sanitize → normalize → auto-fix → validate** 管道

**为什么？**
LLM 不是编译器。它会：
- 用 `func` 代替 `function`
- 忘记填 `complexity` 字段
- 把 weight 写成字符串 `"0.8"` 而不是数字 `0.8`
- 产生悬挂边（引用不存在的节点）

这些都需要自动修复而不是直接报错退出——**部分结果总比没有结果好**。
